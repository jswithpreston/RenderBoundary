React Server Components represent the most significant architectural shift in React since hooks. But most explanations stop at "they run on the server." Let's go deeper.

## The mental model

Server Components aren't a rendering strategy — they're a **component type**. The distinction matters. SSR renders your entire tree to HTML on the server. RSC renders *some* components on the server and sends their output as a serialized tree, not HTML.

The output of a Server Component is a **React element tree encoded as JSON-like payload** — what the React team calls the "RSC wire format."

## The wire format

When a Server Component renders, the output looks something like this:

```json
{
  "type": "div",
  "props": {
    "className": "container",
    "children": [
      {
        "type": "$Lclient-component-id",
        "props": { "count": 42 }
      },
      "Some static text"
    ]
  }
}
```

Notice `$Lclient-component-id`. That's a **reference** to a Client Component. The server doesn't render it — it serializes a pointer. The client bundle knows how to resolve that reference and hydrate the component.

## Serialization boundaries

This is where most people get confused. The boundary between Server and Client Components is a **serialization boundary**. Everything that crosses it must be serializable:

```tsx
// Server Component
export default function Page() {
  const data = await db.query("SELECT * FROM posts");

  // `data` is serializable (plain objects), so this works
  return <ClientList items={data} />;
}
```

What can't cross the boundary:

- Functions (closures capture scope that can't be serialized)
- Class instances
- Symbols
- Anything with circular references

The `"use client"` directive doesn't mean "this component runs on the client." It means "this is the entry point into a client subtree." Everything imported by a `"use client"` module stays in the client bundle.

## Streaming and Suspense

RSC integrates with Suspense at the protocol level. When a Server Component suspends (e.g., awaiting data), the framework streams the rest of the tree and fills in the suspended chunk later:

```
// Stream chunk 1: the shell
0:["$","div",null,{"children":["$","$Suspense",null,{"fallback":"Loading...","children":"$L1"}]}]

// Stream chunk 2: the resolved content (arrives later)
1:["$","article",null,{"children":"Post content here"}]
```

The `$L1` is a lazy reference. When chunk `1:` arrives, React resolves it and updates the tree — no full re-render, no layout shift if you've set up your fallbacks correctly.

## The compilation step

Frameworks like Next.js use a compiler to:

1. **Split the module graph** at `"use client"` boundaries
2. **Generate client references** for each Client Component
3. **Bundle server-only code** separately (never shipped to the browser)

```tsx title="What the compiler sees"
// app/page.tsx — Server Component (no directive)
import { Counter } from "./counter";

export default function Page() {
  return <Counter initialCount={0} />;
}

// counter.tsx — Client Component
"use client";
import { useState } from "react";

export function Counter({ initialCount }: { initialCount: number }) {
  const [count, setCount] = useState(initialCount);
  return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
}
```

The compiler transforms the `Counter` import in `Page` into a client reference. The server never executes `useState` — it just emits a reference like `{"$type": "$Lcounter", "props": {"initialCount": 0}}`.

## Practical implications

### Data fetching moves to the component

No more `getServerSideProps` or loaders. Fetch where you render:

```tsx
async function PostList() {
  const posts = await db.posts.findMany();
  return (
    <ul>
      {posts.map(post => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  );
}
```

This colocation of data and UI is the core win. The component is self-contained.

### Bundle size drops

Server Components don't ship JavaScript to the client. A markdown renderer, syntax highlighter, date formatting library — if they only run in Server Components, they're zero-cost to the client bundle.

### Composition patterns change

You can pass Server Components as `children` to Client Components:

```tsx
// Server Component
export default function Layout() {
  return (
    <ClientSidebar>
      <ServerContent />  {/* This works! */}
    </ClientSidebar>
  );
}
```

This works because `children` is a serialized React element by the time the Client Component receives it. The Client Component doesn't need to know or care that it was server-rendered.

## What RSC is not

- **Not a replacement for SSR.** RSC and SSR are complementary. RSC determines *what* renders where. SSR determines *when* the initial HTML is generated.
- **Not a caching layer.** Though Server Components are cacheable, that's a framework concern, not an RSC primitive.
- **Not required.** You can build entire apps with Client Components only. RSC is an optimization boundary.

The real insight is that RSC gives React a **module-level code splitting primitive** tied to the component model. That's novel. No other framework does it this way.
